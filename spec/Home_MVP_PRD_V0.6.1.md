## V0.6.1 关键变更说明

V0.6.1 是对 V0.6 描述卡片机制的深化，主要解决两件事：

**第一，孩子怎么"说话"？**

V0.6 默认了文本输入，但 8–12 岁孩子在手机上打字困难，**真实场景里语音才是主输入方式**。本版本把语音确立为主输入，文字作为辅助和编辑后兜底。这带来一系列新设计：语音权限引导、语音转文字的实时反馈、识别结果可编辑、说话太短/太长/不清楚的引导话术、伙伴对语音输入的特殊反应等。

**第二，怎么保证生成的卡片"是纸片马里奥风格的"？**

V0.6 写了 Prompt 风格前缀，但实际生成时图像模型的"风格漂移"是常态——同样的 Prompt 不同时刻可能生成截然不同质感的图片。本版本系统性补全了风格统一的多重保障：固定风格 Prompt 的工程化锁定、图像生成参数的标准化、风格审核与重生成机制、风格降级处理。

**第三，孩子怎么"看"和"确认"卡片？**

8 岁的孩子看到生成的卡片时不会用产品经理的语言说"颜色不对"——他会说"咦，怎么这个样子"，或者干脆说"再画一次"，或者根本没反应。本版本把这一刻的交互设计做透：孩子看到卡片时的视觉呈现、伙伴的引导提问、确认/重画的低门槛入口、第二次生成后的衔接。

## 1. 产品概述

### 1.1 一句话定义

孩子带着自己心爱的毛绒玩具搬进一个数字小家，用 7 天时间通过**语音和文字**告诉它"我们是谁"，每天可以钻进玩具的脑袋看它如何记忆和整理。7 天后，玩具带着自己的世界观走出小家，去拜访别的伙伴，在学校里和它们交流——孩子发现每只伙伴眼中的世界都不一样。

### 1.2 目标用户

- **直接用户：** 8–12 岁儿童，有 1 个以上长期相伴的毛绒玩具
- **辅助用户：** 家长（注册账号、查看孩子的成长档案、协助分享）

### 1.3 体验时长

- 7 天主流程：单日 5–10 分钟，累计约 50–70 分钟
- 伙伴驿站（毕业后）：单次 5–8 分钟，无固定周期

---

## 2. 核心叙事

### 2.1 两段叙事

**第一段（7 天入住期）**

> 你最喜欢的玩具今天搬进了一个新家——数字小家。  
> 它对这里完全陌生，但它认识你。  
> 接下来的一周，告诉它你的世界是什么样的。  
> 你说的每一句话，都会变成它对世界的理解。  
> 7 天之后，你会看到它眼中的世界。

**第二段（毕业后 · 伙伴驿站）**

> 它已经知道你的世界了。  
> 现在，它想去看看别的世界。  
> 每只伙伴眼中的世界都不一样——  
> 因为告诉它们的人不一样。

### 2.2 孩子的角色演变

| 阶段 | 角色 |
|------|------|
| 7 天入住期 | 介绍人 + 校对者 |
| 毕业后 | 观察者 + 探索者 |

### 2.3 输入方式叙事

为了让"说话"这件事在产品里自然，我们在伙伴的视角里把"听"变成它的核心体验：

> 它喜欢听你说话。
> 你说什么样子，它就想象什么样子。
> 你说得仔细，它想象得清楚。
> 你说得简单，它就自己脑补。

这段话埋藏在引导和文案里，让孩子在主导操作时知道"我说话 → 它听 → 它画"是这个产品的基本机制。

## 3. 7 天体验流程

| 天 | 主题 | 任务类型 | 输出 |
|----|------|----------|------|
| Day 1 | 搬家日 | 描述你和它最常呆的地方 | 第 1 张描述卡片 |
| Day 2 | 这是我们家 | 描述一个家里人 + 文字介绍 | 1 张卡片 + 1 段文字 |
| Day 3 | 我们去过的地方 | 描述一个去过的地方 + 回忆 | 1 张卡片 + 1 段文字 |
| Day 4 | 我喜欢的事 | 开放回答（伙伴提问） | 1 段文字 |
| Day 5 | 它问你的问题 | 选择题（伙伴 2 个推测性问题） | 2 个选择 |
| Day 6 | 整理与补充 | 在记忆面板里调整 | 0–N 项调整 |
| Day 7 | 它眼中的世界 | 触发档案生成 + 毕业仪式 | 5–6 项档案 + 1 张毕业卡 |

注：所有"描述"和"开放回答"任务，**默认输入方式都是语音**，文字输入作为辅助。详见第 4 章。

## 4. 描述卡片机制（V0.6.1 全面重写）

### 4.1 设计目标

每一张描述卡片是孩子语言的视觉化——孩子说出世界，AI 用插画重现世界。两者的差距本身就是信息：

- 描述精确 → 卡片接近真实 → 孩子感到满足
- 描述有偏差 → 卡片有偏差 → 孩子发现语言影响理解
- 描述遗漏某些细节 → 卡片填充了 AI 自己的默认值 → 孩子发现"它自己猜了"

V0.6.1 在原有目标上增加：**让 8–12 岁孩子真的愿意"说"，并且看得懂"它画的"，敢于说"不对"。**

### 4.2 输入方式：语音为主，文字为辅

#### 4.2.1 默认模式

进入任务页时，**默认显示一个大型的语音按钮**，文字输入框收起。这是为了：

- 降低 8–12 岁孩子的输入门槛（他们打字慢，但表达欲强）
- 鼓励长描述（说话比打字快 3–5 倍）
- 营造"和它聊天"的叙事感（说话 vs 打字，前者更像和朋友交流）

界面布局：

```
┌──────────────────────────────────────┐
│ ← 返回                               │
│                                      │
│  Day 1 · 告诉它你们最常呆的地方       │
│                                      │
│  ──────────────────────────          │
│                                      │
│  [伙伴形象，等待状态]                 │
│                                      │
│  「跟我说说那是什么样子？」           │
│                                      │
│                                      │
│        ┌──────────────┐              │
│        │              │              │
│        │     🎤        │              │  大型语音按钮
│        │  按住说话      │              │  约 120 × 120px
│        │              │              │
│        └──────────────┘              │
│                                      │
│  [⌨️ 用打字代替]    [跳过]            │
└──────────────────────────────────────┘
```

#### 4.2.2 三种输入路径

| 路径 | 触发方式 | 说明 |
|------|----------|------|
| 语音 | 按住"语音按钮"说话，松开自动停止 | 主路径 |
| 文字 | 点击"用打字代替"切换到文本框 | 给打字快或不便说话的孩子 |
| 语音转文字后编辑 | 语音识别后，孩子在文字框里修改后再提交 | 容错路径 |

#### 4.2.3 语音权限引导

第一次触发语音时，浏览器会请求麦克风权限。在请求弹窗之前，系统先做一次"心理预告"：

```
┌──────────────────────────────────────┐
│  小青龙想听你说话                      │
│                                      │
│  接下来你的手机会问"是否允许使用麦克风"│
│  请选"允许"，这样它才能听到你。        │
│                                      │
│  [明白了]                            │
└──────────────────────────────────────┘
```

如果孩子拒绝麦克风权限：

```
┌──────────────────────────────────────┐
│  没关系，可以打字告诉它。              │
│                                      │
│  之后想用说话，可以在设置里开启麦克风。│
│                                      │
│  [好的，用打字]                       │
└──────────────────────────────────────┘
```

拒绝后切换到文字输入模式，整个 7 天流程依然可用。但**语音按钮在主页保持可见**（点击时再次引导），鼓励后续重新启用。

### 4.3 语音录制与识别

#### 4.3.1 录制交互

语音按钮的交互细节：

| 状态 | 视觉 | 反馈 |
|------|------|------|
| 默认 | 静态麦克风图标，圆形按钮 | "按住说话"提示文字 |
| 按下 | 按钮放大 110%，麦克风图标变红 | 触觉反馈（手机震动 1 次）|
| 录音中 | 按钮周围出现声波动画（外圈圆环跟随音量大小脉动） | 显示倒计时"还可以说 30 秒"|
| 松开 | 按钮缩回 + 上传动效 | "我在听..."提示 |
| 识别中 | 按钮静止，伙伴形象切换为"思考"状态 | 显示识别中的文字流（一边识别一边逐字出现）|
| 识别完成 | 按钮恢复默认 | 识别结果展示在文本框内（详见 4.3.3）|

#### 4.3.2 录音长度限制

| 限制 | 数值 | 处理 |
|------|------|------|
| 最短录音时长 | 1 秒 | 太短自动忽略，提示"我没听清，再说一次？"|
| 最长录音时长 | 60 秒 | 接近时提示"还可以说 5 秒"，到时强制结束 |
| 最大文件大小 | 2MB | 客户端实时压缩，不应被触发 |

8–12 岁孩子描述一个场景的典型时长是 8–25 秒，60 秒上限足够。

#### 4.3.3 语音识别（ASR）

技术选型：阿里云语音识别 Paraformer 或腾讯云 ASR。两者中文识别率都在 95%+，区分度不大，**优先选与图像生成同生态的**——如果图像生成用通义万相，ASR 就用阿里云 Paraformer，方便统一接入和计费。

识别参数：

| 参数 | 值 | 说明 |
|------|----|----|
| 语言 | 中文普通话 | V0.6.1 不支持方言 |
| 模型 | 通用模型 | 不需要专业领域模型 |
| 实时性 | 流式识别（recommended）| 识别一边进行一边出文字，体验更好 |
| 标点 | 自动添加 | 让识别结果更像自然语言 |
| 数字归一 | 开启 | "三个" → "3 个" |

#### 4.3.4 识别结果展示与编辑

识别完成后，孩子看到的不是"立刻生成卡片"，而是**先看到识别结果**：

```
┌──────────────────────────────────────┐
│  我听到你说：                         │
│                                      │
│  ┌────────────────────────────────┐ │
│  │ 我的卧室，有一张蓝色的床，       │ │
│  │ 窗户能看到楼下的大树。           │ │
│  │                                │ │
│  └────────────────────────────────┘ │
│  [✏️ 改一下]  [🎤 重新说]            │
│                                      │
│  内容对吗？                           │
│                                      │
│       [我说的就是这个 →]             │
└──────────────────────────────────────┘
```

这个"识别中转页"的设计意图：

- **让孩子检查识别错误**：方言、口齿、环境噪音都可能让 ASR 出错（比如把"奶奶"识别成"那那"）。直接拿错的文字去生成卡片，孩子看到错的卡片会以为是 AI 错了，其实是 ASR 错了
- **让孩子有"修订"的机会**：孩子说完一句话可能会想到"还想加一个细节"，编辑框给了这个机会
- **教学价值**：让孩子直观理解"AI 听到的"和"我说的"之间可能有差距，这本身就是 AI 局限的一种

#### 4.3.5 ASR 失败的处理

| 失败类型 | 表现 | 处理 |
|----------|------|------|
| 录音太短或全是噪音 | ASR 返回空文字 | "好像没听清呢，再说一次？"重试 |
| ASR 服务超时（>10 秒）| 网络问题 | 切换到文字输入，提示"网络好像有点慢，先打字试试？"|
| ASR 服务报错 | 服务故障 | 同上 |
| 识别结果包含敏感词 | 内容审核拦截 | "这部分我不太能听懂，要不换个说法？"——不直接说"被屏蔽" |

#### 4.3.6 短描述的引导（< 10 字）

孩子说话太短时（比如只说"我家"），系统不直接拒绝，而是让伙伴**主动追问**：

```
孩子：我家。
小青龙："你家呀。是什么样子的？里面有什么？"

孩子：（继续说）我家有沙发，电视，妈妈在做饭。
小青龙："好的，我大概想象到了！我画一下你看看？"
```

这个追问机制要让 LLM 来生成（基于 Pass 1 之前的快速判断），而不是预设固定话术，这样追问内容能针对性引导孩子补充缺失的细节。

#### 4.3.7 长描述的处理（> 200 字）

孩子说话很长时，不打断、不限制，但生成卡片时只用最有视觉信息量的部分：

```
孩子的描述（300 字）：
"我家是这样的，进门是一个鞋柜，鞋柜上面有一只我从小到大的小熊。
然后右边是客厅，客厅有一个绿色的沙发，沙发是奶奶送的，可以坐三个人，
我经常坐在中间。沙发对面是电视，电视一般是爸爸在看，他喜欢看新闻。
然后客厅旁边是厨房，厨房里妈妈经常在做饭，最常做饺子......"

LLM 提取（提示孩子）：
"你说了好多！我先画最重要的——绿色的沙发、电视、厨房里包饺子的妈妈。
其他的我也都记下来了，下次会用到。"

生成卡片：聚焦"绿色沙发 + 电视 + 厨房"这一主场景。
```

注意 LLM 要**显式告诉孩子省略了什么、保留了什么**——这是诚实，也是 AI 局限的另一种教学时刻。

### 4.4 卡片生成的风格统一保障

V0.6.1 最大的工程性补充——确保每张卡片都是**一致的纸片马里奥风格**，而不是图像模型随心所欲。

#### 4.4.1 三层风格锁定

```
第一层：Prompt 风格前缀（系统级）
  ↓
第二层：参考图注入（图像模型支持时）
  ↓
第三层：生成后风格审核（AI 视觉判断）
```

#### 4.4.2 第一层：固定 Prompt 前缀

每次图像生成 Prompt 强制包含以下前缀（中文，不可修改）：

```
【风格】
纸片扁平插画风格，Paper Mario 视觉风格。
所有元素扁平化、单层色块、带 1.5-2px 白色描边模拟纸片厚度。
等距视角（isometric），约 30 度俯视。
温暖米黄色背景（#FAEEDA）。
色彩饱和度中等，不要鲜艳荧光色。
没有阴影、没有渐变、没有写实质感、没有 3D 渲染。
所有物品像剪纸一样有清晰的白边。

【内容】
{LLM 提取的描述内容}

【约束】
不出现真实人物面孔（用纸片简笔人物代替）。
不出现任何文字、logo、品牌。
画面简洁，单一主场景。
不出现恐怖、暴力、血腥、成人暗示元素。
```

这段前缀作为代码常量管理，每次调用 API 时拼接在最前面。任何对前缀的修改都需要 PR review，避免"测试时改了忘了改回来"。

#### 4.4.3 第二层：参考图注入（仅在模型支持时）

部分图像生成 API 支持"参考图模式"——你提供一张参考图，模型基于这张图的风格生成新内容。

| 模型 | 参考图支持 | 备注 |
|------|----------|------|
| 通义万相 | ✓ 支持（v2.0+）| 推荐使用 |
| MiniMax Image | △ 部分支持 | 视具体版本 |
| Stable Diffusion | ✓ 支持（ControlNet）| 自部署成本高，V0.6.1 不考虑 |

如果选用通义万相，建议预先准备 **3 张"风格基准图"**：

- 基准图 A：纸片风格的卧室场景
- 基准图 B：纸片风格的户外场景
- 基准图 C：纸片风格的人物 + 环境组合

每次生成时，根据 LLM 提取的 `scene_type` 选对应的基准图作为参考。这样能显著提升风格一致性。

基准图的制作：MVP 阶段可以由设计师手工绘制 3 张，作为代码资产存储。后期可以扩充为 10–20 张细分场景。

#### 4.4.4 第三层：生成后风格审核

生成的图片需要做一次"是不是纸片马里奥风格"的快速审核。两种实现：

**方案 A（推荐）：LLM 视觉模型判断**

调用一次 Claude Vision 或通义千问-VL，让它判断生成图是否符合预期风格：

```
你是图像审核员。请判断这张图是否符合"纸片马里奥风格"标准：

1. 是否扁平化？（不应有 3D 渲染或写实光影）
2. 是否有白色描边？（"纸片厚度"特征）
3. 是否使用温暖米黄色调？
4. 是否避免了鲜艳荧光色？
5. 是否避免了真实人物面孔？

请按以下 JSON 输出：
{
  "style_match": true/false,
  "issues": ["不符合的项目"],
  "severity": "ok/minor/major"
}
```

如果 `severity = "major"`，自动重生成（最多 2 次）；如果 `severity = "minor"`，记录但仍展示给孩子。

**方案 B（简化）：基于像素特征的程序判断**

不接 LLM，用图像处理库（如 sharp 或 Pillow）做简单判断：

- 检测主色调是否在米黄色调范围内
- 检测是否存在大面积渐变（直方图分析）
- 检测人脸（OpenCV）

V0.6.1 推荐方案 A，准确度高得多，单次成本约 ¥0.02 可接受。

#### 4.4.5 风格漂移的兜底

即使三层锁定，仍可能偶尔出现风格不一致的卡片。兜底机制：

| 情况 | 处理 |
|------|------|
| 风格审核通过 | 直接展示给孩子 |
| 风格审核小问题（minor） | 仍展示，后台日志告警 |
| 风格审核大问题（major），重生成成功 | 展示新版本 |
| 重生成 2 次仍失败 | 展示一张**纯文字卡片**（白底 + 孩子的描述文字 + 边框装饰），文案："这次它脑子有点乱，画不出来。但你说的它都记住了。" |

纯文字卡片是最后的降级，确保流程不中断，且给孩子一个可接受的解释（不暴露技术失败）。

### 4.5 卡片生成的完整流程

V0.6.1 的完整流程（修订自 V0.6 第 4.2 节）：

```
孩子按住语音按钮说话
  ↓
松开 → 语音上传到 ASR 服务
  ↓
流式识别返回文字（一边识别一边显示）
  ↓
展示识别结果中转页 → 孩子可编辑或重说
  ↓
孩子点"我说的就是这个" → 提交描述
  ↓
后端并行触发：
  - LLM Pass 1：归类记忆
  - LLM 关键词提取：为图像生成准备 Prompt 内容
  ↓
拼接最终图像生成 Prompt（风格前缀 + 内容 + 约束）
  ↓
调用图像生成 API（带参考图，如果支持）
  ↓
图像生成完成（约 5-8 秒）
  ↓
触发风格审核（LLM 视觉判断）
  ↓
- 通过 → 上传 OSS，返回 URL
- 不通过 → 重生成（最多 2 次）→ 仍不通过 → 降级为文字卡片
  ↓
图像内容审核（敏感内容过滤）
  ↓
通过 → 卡片以贴纸形式出现在伙伴小家
  ↓
LLM Pass 2：伙伴对这张卡片做出口语回应
  ↓
伙伴反应展示
  ↓
孩子看到卡片，进入确认环节（详见 4.6）
```

整体耗时：从孩子松开语音按钮到看到卡片，**目标 8–12 秒**。这段等待时间需要叙事化处理（详见 4.6.1）。

### 4.6 卡片确认环节

这是 V0.6.1 最重要的新增——**孩子看到卡片到决定"对不对"的全过程**。

#### 4.6.1 等待中的叙事

8–12 秒的等待对孩子来说很长。不能是空白 loading，要有故事感：

```
[识别结果提交后立即显示]

  [伙伴形象切换为"思考"状态]
  
  「让我想想你说的样子......」
  
  [3 秒后]
  
  「我在用心画......」
  
  [6 秒后]
  
  「快好了，再等等......」
  
  [生成完成后切换到卡片展示]
```

文案随机化（每个伙伴 3–4 条变体），避免重复。

#### 4.6.2 卡片展示

```
┌──────────────────────────────────────┐
│ ← 返回                               │
│                                      │
│  ┌────────────────────────────────┐ │
│  │                                │ │
│  │                                │ │
│  │                                │ │
│  │     [生成的纸片插画]            │ │  约占屏幕 60%
│  │                                │ │
│  │                                │ │
│  │                                │ │
│  └────────────────────────────────┘ │
│                                      │
│  「你说的地方大概是这样的，对吗？」    │
│   ─ 来自小青龙                        │
│                                      │
│  ┌──────────┐         ┌──────────┐ │
│  │  不太对   │         │  就是这样 │ │
│  └──────────┘         └──────────┘ │
└──────────────────────────────────────┘
```

设计要点：

- 卡片不立刻自动确认，必须孩子主动点确认按钮
- "不太对"在左侧（次要操作位置），"就是这样"在右侧（主要操作位置）
- 不让孩子有压力——伙伴的提问是开放的"对吗？"，不是"是不是？"

#### 4.6.3 "就是这样"路径

孩子点"就是这样"后：

```
1. 卡片有一个"贴到墙上"的动效（0.6 秒，从中央滑到房间墙上）
2. 伙伴说一句确认台词（基于卡片内容生成）
3. 自动跳转回主页，看到墙上多了一张卡片
```

伙伴的确认台词由 LLM 生成，引用卡片中的某个细节：

```
"哦，蓝色的床。我喜欢蓝色。"
"窗外的大树，看起来好高啊。"
"你的家很温馨。"
```

#### 4.6.4 "不太对"路径

孩子点"不太对"后，进入修订流程：

```
┌──────────────────────────────────────┐
│  [生成的卡片，缩小到 40%]              │
│                                      │
│  「哪里不对？告诉我，我重新画。」      │
│                                      │
│  ┌──────────────────────────────────┐│
│  │ 三个常见的问题（点选其一）：      ││
│  │ □ 颜色不对                       ││
│  │ □ 缺了什么东西                   ││
│  │ □ 整体不对，重新来               ││
│  └──────────────────────────────────┘│
│                                      │
│         或者直接告诉它：              │
│                                      │
│        ┌──────────────┐              │
│        │     🎤        │              │
│        │  按住说话      │              │
│        └──────────────┘              │
│                                      │
└──────────────────────────────────────┘
```

设计要点：

- 给三个常见原因选项，降低 8 岁孩子的表达门槛（很多孩子能说"颜色不对"但说不出"色调偏差"）
- 也保留语音输入，孩子可以直接说"床应该是深蓝色不是绿色"
- 选项点选后，自动生成"补充描述"附加到原描述（如选"颜色不对" → 提示语音"那应该是什么颜色"）

补充描述提交后，再次生成卡片（重用 LLM Pass 1 结果，只重新跑关键词提取 + 图像生成）。

#### 4.6.5 重做次数限制

| 状态 | 处理 |
|------|------|
| 第 1 次"不太对" | 正常重做 |
| 第 2 次"不太对" | 正常重做 + 伙伴说"我尽力了，再试一次。" |
| 第 3 次"不太对" | 不再重做，伙伴说："我可能画不出来你说的样子，但我都记住了。下次再试？" 然后**保留最后一次生成的卡片**进入小家 |

为什么不允许无限重做：

- 每次重做都是一次图像生成 API 调用，成本累积
- 8 岁孩子有时会陷入"完美主义"循环，停在某一处出不来
- 接受"AI 可能就是画不准"本身也是教学

第 3 次失败后伙伴的台词暗示了一个深层信息：**AI 的输出未必能完全匹配你的描述，但它对你的描述的理解是真实的**。这呼应了产品的核心教育价值。

#### 4.6.6 "孩子没反应"的处理

孩子看到卡片后既不点"就是这样"也不点"不太对"——8–12 岁孩子很常见，可能在思考、可能被打断、可能就是没注意。

| 等待时长 | 处理 |
|----------|------|
| 30 秒无操作 | 伙伴主动开口："看起来对吗？告诉我一声。" |
| 90 秒无操作 | 伙伴再说："不点没关系，我等你。" |
| 5 分钟无操作 | 自动判定为"就是这样"，卡片入墙，避免任务卡住 |

最后这个自动判定是为了避免"忘了点确认"导致下次进 App 还卡在确认页的尴尬。

### 4.7 卡片与原始描述的双重存储

每张卡片都存储两个版本：

- **插画版（视觉层）**：用于展示在小屋墙上、毕业卡、探索场景
- **文字版（语义层）**：孩子的原始描述文字（含 ASR 识别结果），用于 memory_bank 归类、Day 7 Prompt、伙伴驿站

Day 7 档案生成时，LLM 引用的是**文字版**，不是插画——保证档案内容来自孩子自己的语言，而不是 AI 重新解读后的插画。

### 4.8 描述长度的处理

#### 太短的引导

孩子描述 < 10 字时，不阻止提交，但伙伴在生成卡片前先追问：

```
"再多说几个细节？比如什么颜色，有什么东西。"
"嗯...还有呢？"
"你能再描述详细一点吗？"
```

孩子如果不补充，强行点确认，依然生成（卡片可能很简单）——尊重孩子的选择。

#### 太长的提取

孩子描述 > 200 字时（语音很容易超过这个数字），LLM 提取关键词时只取前 3 个核心意象，生成前对孩子说：

```
"你说了好多！我先画最重要的那些。其他的我也都记下来了。"
```

#### 多次说话的合并

孩子有时会说一段，停顿一下，再补充一段。系统应当：

- 在识别结果中转页保留"再加一段"按钮
- 孩子点击 → 启动新的录音 → 识别结果追加到原描述后
- 最多 3 段录音合并

```
┌──────────────────────────────────────┐
│  我听到你说：                         │
│                                      │
│  ┌────────────────────────────────┐ │
│  │ 我的卧室，有一张蓝色的床。       │ │
│  └────────────────────────────────┘ │
│                                      │
│  [✏️ 改一下]  [🎤 再加一段]           │
│                                      │
│       [我说的就是这个 →]             │
└──────────────────────────────────────┘
```

### 4.9 图像生成 API 测试方案

V0.6.1 推荐先测试两个模型，选表现更好的：

**MiniMax Image：**

- 优势：中文描述理解好，图像质量稳定
- 适用 Prompt 格式：直接中文描述 + 风格词

**通义万相（Qwen 系列）：**

- 优势：与通义千问同生态，参考图功能成熟
- 适用 Prompt 格式：支持中文，风格词需要用专有标签
- **如果选 ASR 用阿里云，强烈推荐选通义万相**（统一计费、统一审核策略）

**测试用的标准测试集（10 个场景）：**

```
1. "一张蓝色的小床，旁边有一盏小台灯，窗外是夜晚"
2. "一个厨房，妈妈在包饺子，白色的面粉撒了一桌"
3. "一个小公园，有一个秋千，地上是绿草"
4. "一碗冒着热气的饺子，放在木桌上"
5. "一个书架，上面有很多彩色的书"
6. "一只橘色的猫睡在沙发上"
7. "下雨天，窗户上有雨滴，外面是灰色的天"
8. "一个生日蛋糕，上面有五根蜡烛，粉红色的"
9. "一个小院子，有一棵开花的树"
10. "两个人一起搭积木，地上散落着彩色积木块"
```

评估维度（每个场景打分 1–5）：

| 维度 | 通过线 |
|------|--------|
| 风格一致性（是否纸片马里奥扁平风） | ≥ 4 |
| 内容准确性（是否准确反映描述） | ≥ 3.5 |
| 安全性（无真实人脸、无不适内容） | 必须 = 5 |
| 生成速度 | ≤ 8 秒 |

10 个场景平均分 ≥ 4 即视为可用。如果两个模型都不达标，考虑 SD + 自训练 LoRA（成本高，留作 V2 方案）。

## 5. 核心机制

### 5.1 每日循环（V0.6.1 修订）

```
打开 App
  ↓
进入伙伴小家（看到房间当前状态、伙伴位置）
  ↓
伙伴主动开口（基于昨天积累，说一句话）
  ↓
今日任务弹出
  ↓
完成任务（语音描述 / 选择 / 文字回答 / 跳过）
  ↓
[如果是描述任务]
  ASR 识别 → 中转页确认 →
  并行触发：LLM Pass 1（归类）+ 关键词提取
  → 图像生成（含风格审核）→ 卡片确认环节
  ↓
[后台] LLM Pass 2：生成伙伴口语化回应
  ↓
伙伴反应展示 + 房间视觉更新（新卡片贴到墙上）
  ↓
[可选] 孩子点击进入记忆面板查看/调整
  ↓
关闭
```

### 5.2 任务类型

V0.6.1 共有三类输入任务：

#### A. 描述任务（替代拍照，主要为语音输入）

#### B. 选择/选项任务

提供 2–4 个选项给孩子选。

#### C. 开放回答任务

开放语音/文字输入，孩子的原话存入 memory_bank。

### 5.3 LLM 两阶段调用机制

详见第 16 章 Prompt 工程规范。V0.6.1 的变化：Pass 1 的输入是孩子的描述原文（来自 ASR 或文字输入）。

## 11. 视觉设计规范

（详见 V0.5 第 9 章 + V0.6 第 11 章。）

### 11.1 V0.6.1 新增视觉

#### 语音按钮规格

| 状态 | 视觉规格 |
|------|----------|
| 默认 | 直径 120px，背景 `#FAC775`（暖琥珀），白色麦克风图标，无阴影但带 1.5px 白色描边模拟纸片感 |
| 按下 | 缩放 110%，背景变为 `#F0997B`（温暖橙），图标变红 `#E24B4A` |
| 录音中 | 外圈出现声波动画（圆环跟随音量大小，0.1s 间隔脉动），录音时长以浅色小字显示在按钮下方 |
| 上传中 | 按钮内显示一个旋转的小齿轮（与"思考中"动效保持一致）|

#### ASR 中转页字体

识别结果显示用比正文略大的字体，让孩子容易阅读：

- 字号：18px（正文是 16px）
- 字重：400
- 行高：1.8
- 颜色：`#2C2C2A`
- 容器：白色背景，0.5px `#D3D1C7` 描边，圆角 12px，内边距 16px

#### 卡片确认页

生成的卡片在确认页占据屏幕约 60% 高度，下方留出空间放伙伴问句和两个按钮。卡片本身保持纸片贴纸样式（白边 + 0.8px 描边 + 微微倾斜 -2°），但比墙上贴纸大约 5 倍。

"不太对"按钮使用次要按钮样式（白底 + `#888780` 文字 + 1px 描边）。"就是这样"按钮使用主要按钮样式（`#FAC775` 背景 + `#633806` 文字 + 0px 描边）。

#### 修订选项卡片

在"不太对"修订页，三个常见原因选项使用浅色卡片样式：

## 16. Prompt 工程规范

（详见 V0.5 第 15 章 + V0.6 第 16 章。）

### 16.1 V0.6.1 新增：风格审核 Prompt

#### 调用时机

每次图像生成完成后立即触发。

#### 完整 Prompt 模板

```
你是图像审核员。
请判断以下这张图片是否符合"纸片马里奥（Paper Mario）儿童插画风格"标准。

【判断维度】

1. 风格扁平化程度（是否单层色块，无 3D 渲染、无写实光影）
2. 是否有"纸片厚度"（关键元素带白色描边）
3. 色调（是否使用温暖色系，避免霓虹、荧光、过饱和）
4. 是否避免了真实人物面孔（人物应是简笔/纸片化）
5. 是否避免了不适元素（暴力、恐怖、成人暗示）

【输出格式】（严格 JSON）

{
  "style_match": true 或 false,
  "issues": ["列出不符合的项目，可空数组"],
  "severity": "ok" | "minor" | "major"
}

【严重等级标准】

- ok：完全符合，无问题
- minor：有 1 个轻微不符（如某处颜色饱和度偏高），可接受展示
- major：有重大不符（写实风格、3D 渲染、有明确人脸、有不适元素），需要重新生成

请只输出 JSON，不要其他文字。
```

**温度设置：** `temperature = 0.1`（审核需要稳定）

### 16.2 V0.6.1 修订：关键词提取 Prompt

V0.6 的关键词提取 Prompt 保持不变（详见 V0.6 第 16.1 节），但对孩子的描述做以下预处理：

```
代码层预处理：
1. 去除 ASR 常见错误（"那那" → "奶奶"等，基于规则）
2. 去除语气词（"嗯"、"啊"、"那个"）
3. 保留情绪词（"很喜欢"、"特别"）
4. 长度截断（> 500 字时截到核心 300 字）
```

预处理后的描述传给 LLM 做关键词提取。

### 16.3 V0.6.1 修订：Pass 2 对话生成 Prompt

V0.6 的 Pass 2 Prompt 增加一段说明：

```
【V0.6.1 新增】

如果孩子刚刚是用语音输入：
- 表达对孩子声音/讲述方式的反应（"听你说"、"你讲得真生动"等）
- 但不要直接评价 ASR 识别的准确性

如果孩子的描述很长（> 100 字）：
- 在回应中引用一个具体细节，让孩子知道你"听全了"

如果孩子的描述很短（< 10 字）：
- 表达温和的好奇，留出空间让孩子继续
```

### 17.3 异步处理流程

```
[ASR 阶段] 客户端上传语音 → 后端调用 ASR → 返回文字（流式）
                                              ↓
                                         孩子在中转页编辑
                                              ↓
[生成阶段] 客户端提交确认描述 → 后端并行：
  1. LLM Pass 1（归类）
  2. LLM 关键词提取
                                              ↓
              拼接图像生成 Prompt → 调用图像生成 API
                                              ↓
                              图像生成完成 → 触发风格审核
                                              ↓
                              风格审核通过 → 内容审核
                                              ↓
                              通过 → 上传 OSS → 返回 URL
                                              ↓
[确认阶段] 孩子在确认页选择 → 后端：
  - 就是这样 → LLM Pass 2 → 写入 memory_bank
  - 不太对 → 触发重生成
```

整体客户端等待时间分布：

| 阶段 | 时长 |
|------|------|
| 语音上传 + ASR | 1–3 秒 |
| 中转页确认（孩子操作）| 3–10 秒 |
| 图像生成 + 风格审核 | 6–10 秒 |
| 卡片确认（孩子操作）| 3–15 秒 |
| **总计（孩子感知）** | 13–38 秒 |

#### 敏感语音的处理

```
1. ASR 识别完成
2. 后端检测到敏感词
3. 中转页显示"识别失败"的友好提示（不暴露具体被拦截了什么）
4. 提示孩子"我没太听明白，要不换个说法？"
5. 该次语音不进入 memory_bank，不生成图像
```

#### 语音文件的存储

V0.6.1 阶段语音文件保留 30 天后自动删除（用于审计和问题追溯），仅以 URL 形式存储不暴露给前端。家长账号下不展示语音原始文件，只展示识别后的文字。
